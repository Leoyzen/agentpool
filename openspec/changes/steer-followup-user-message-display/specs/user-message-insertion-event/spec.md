## ADDED Requirements

### Requirement: UserMessageInsertedEvent carries inserted user message content

The system SHALL define a `UserMessageInsertedEvent` as a frozen dataclass in `RichAgentStreamEvent` union. The event SHALL carry: `session_id` (str), `message_id` (str, unique per insertion), `content` (`str | list[Any]`, the user's text or multi-modal content), `delivery` (Literal["initial", "steer", "followup"]), `source` (Literal["protocol", "background_task", "internal"], default "protocol"), `timestamp` (float, `time.time()` default factory).

The event SHALL be added to the `RichAgentStreamEvent` PEP 695 `type` statement.

The event SHALL bypass CommChannel — it SHALL NOT be journaled or replayed. It SHALL be published directly to `EventBus` via `event_bus.publish()`, if EventBus is available.

#### Scenario: Event fields populated for steer message
- **WHEN** a steer message is routed through `_route_message()` with `priority="asap"`
- **THEN** a `UserMessageInsertedEvent` is published with `delivery="steer"`, `content` set to the steer text, and a unique `message_id`
- **AND** the event is published to `EventBus` via `event_bus.publish(session_id, event)`, if EventBus is available

#### Scenario: Event fields populated for followup message
- **WHEN** a followup message is routed through `_route_message()` with `priority="when_idle"`
- **THEN** a `UserMessageInsertedEvent` is published with `delivery="followup"`, `content` set to the followup text, and a unique `message_id`

#### Scenario: Event fields populated for initial message
- **WHEN** an initial message starts a new run on an idle session
- **THEN** a `UserMessageInsertedEvent` is published with `delivery="initial"`, `content` set to the prompt text, and a unique `message_id`

#### Scenario: Event supports multi-modal content
- **WHEN** a message with multi-modal content (`list[Any]` containing text and image blocks) is routed
- **THEN** the `content` field of `UserMessageInsertedEvent` carries the `list[Any]` payload
- **AND** protocol handlers convert each element to the appropriate content block type

#### Scenario: Event not journaled
- **WHEN** `UserMessageInsertedEvent` is published to `EventBus`
- **THEN** the event SHALL NOT pass through `CommChannel.publish()`
- **AND** the event SHALL NOT be written to `Journal.append()` or `Journal.upsert()`
- **AND** on crash recovery replay, the event SHALL NOT be replayed

### Requirement: SessionController._route_message() publishes UserMessageInsertedEvent

`SessionController._route_message()` SHALL publish `UserMessageInsertedEvent` to `EventBus` before routing the message, if EventBus is available. The `delivery` field SHALL be determined by session state and priority: `"initial"` for idle sessions, `"steer"` for busy sessions with `priority="asap"`, `"followup"` for busy sessions with `priority="when_idle"`.

The `message_id` SHALL be generated as `str(uuid.uuid4())` and SHALL be consistent with any `message_id` generated by the protocol handler for deduplication. If the protocol handler passes a `message_id` via `send_message(message_id=mid)`, `_route_message()` SHALL use it.

#### Scenario: Steer message publishes event
- **WHEN** `_route_message()` is called with `priority="asap"` on a busy session
- **THEN** `UserMessageInsertedEvent(delivery="steer")` is published to `EventBus`, if EventBus is available
- **AND** the event is published before `run.steer(content)` is called

#### Scenario: Followup message publishes event
- **WHEN** `_route_message()` is called with `priority="when_idle"` on a busy session
- **THEN** `UserMessageInsertedEvent(delivery="followup")` is published to `EventBus`, if EventBus is available
- **AND** the event is published before `session.prompt_queue.put_nowait(content)` is called

#### Scenario: Initial message publishes event
- **WHEN** `_route_message()` is called on an idle session
- **THEN** `UserMessageInsertedEvent(delivery="initial")` is published to `EventBus`, if EventBus is available
- **AND** the event is published before `_start_run_handle()` is called

#### Scenario: Standalone execution without EventBus
- **WHEN** `_route_message()` is called and no EventBus is available (standalone `agent.run()`)
- **THEN** no `UserMessageInsertedEvent` is published
- **AND** the message is still routed normally (no functional impact)

### Requirement: steer_from_background_task() publishes UserMessageInsertedEvent

`SessionController.steer_from_background_task()` SHALL publish `UserMessageInsertedEvent` with `delivery="steer"` and `source="background_task"` before calling `run_handle.steer(content)`, if EventBus is available.

`steer_from_background_task()` is a **synchronous** method. It SHALL use `asyncio.create_task()` for the EventBus publish with `try/except RuntimeError` handling for no-running-loop scenarios.

#### Scenario: Background task steer publishes event
- **WHEN** `steer_from_background_task()` is called
- **THEN** `UserMessageInsertedEvent(delivery="steer", source="background_task")` is published to `EventBus`, if EventBus is available
- **AND** the event is published before `run_handle.steer(content)` is called

#### Scenario: Background task steer with no running loop
- **WHEN** `steer_from_background_task()` is called outside an async context (no running event loop)
- **THEN** the `RuntimeError` from `asyncio.get_running_loop()` is caught
- **AND** emission is silently skipped (the steer still proceeds)

### Requirement: _consume_run() publishes UserMessageInsertedEvent for followup-from-queue

`SessionController._consume_run()` SHALL publish `UserMessageInsertedEvent` with `delivery="followup"` and `source="internal"` when picking up a followup message from `prompt_queue`, if EventBus is available. This covers the gap where `_consume_run()` → `_create_per_prompt_handle()` creates a `RunHandle` directly WITHOUT calling `_route_message()`.

#### Scenario: Followup from prompt_queue publishes event
- **WHEN** `_consume_run()` picks up a message from `prompt_queue` after the current turn
- **THEN** `UserMessageInsertedEvent(delivery="followup", source="internal")` is published to `EventBus`, if EventBus is available
- **AND** the event is published before `_create_per_prompt_handle()` is called

### Requirement: RunHandle.steer() and followup() emit UserMessageInsertedEvent via asyncio.create_task

`RunHandle.steer()` SHALL accept an `emit_user_message: bool = True` parameter. When `True`, it SHALL schedule `UserMessageInsertedEvent` emission via `asyncio.get_running_loop().create_task()` (fire-and-forget). `RunHandle.followup()` SHALL accept an `emit_user_message: bool = False` parameter with the same behavior.

The emission SHALL be best-effort — failure to publish SHALL log a warning but SHALL NOT raise. All `create_task()` call sites SHALL use `try/except RuntimeError` to handle no-running-loop scenarios.

When `emit_user_message=True`, `SystemNotificationEvent` (if implemented in the future via RFC-0056) should default to suppressed for the same message to avoid redundant display.

#### Scenario: steer() with default emit_user_message=True
- **WHEN** `RunHandle.steer(content)` is called with default parameters
- **THEN** a fire-and-forget task is scheduled to publish `UserMessageInsertedEvent(delivery="steer", source="internal")`
- **AND** the steer logic proceeds without waiting for the task

#### Scenario: steer() with emit_user_message=False
- **WHEN** `RunHandle.steer(content, emit_user_message=False)` is called
- **THEN** no `UserMessageInsertedEvent` is scheduled from `steer()`
- **AND** the event may still be published by `_route_message()` if the call went through the protocol handler

#### Scenario: followup() with default emit_user_message=False
- **WHEN** `RunHandle.followup(content)` is called with default parameters
- **THEN** no `UserMessageInsertedEvent` is scheduled from `followup()`
- **AND** the event may still be published by `_consume_run()` for followup-from-queue

#### Scenario: followup() with emit_user_message=True
- **WHEN** `RunHandle.followup(content, emit_user_message=True)` is called
- **THEN** a fire-and-forget task is scheduled to publish `UserMessageInsertedEvent(delivery="followup", source="internal")`

#### Scenario: steer() with no running event loop
- **WHEN** `RunHandle.steer(content, emit_user_message=True)` is called outside an async context
- **THEN** the `RuntimeError` from `asyncio.get_running_loop()` is caught
- **AND** emission is silently skipped (the steer still proceeds)

### Requirement: Protocol handlers deduplicate UserMessageInsertedEvent by message_id via shared dedup set

A shared `message_id` dedup set SHALL be accessible by BOTH the protocol handler emission path AND the `ACPEventConverter`/`EventProcessor` (EventBus path). Protocol handlers SHALL populate the dedup set when they emit user messages. When `UserMessageInsertedEvent` arrives at a protocol handler with a `message_id` already in the set, the handler SHALL skip it (no duplicate display).

The dedup set SHALL be populated by protocol handlers at emission time — `_emit_user_message_chunks()` (ACP) and `message_routes.py` / `session_routes.py` / `opencode_event_bridge.py` (OpenCode) SHALL generate `message_id` FIRST, register it in the shared dedup set, then emit to the frontend and pass the `message_id` to `send_message()` → `_route_message()`.

The set SHALL be bounded — entries SHALL be removed when the session is closed or after a configurable TTL (default: 5 minutes).

#### Scenario: Protocol handler already displayed the message
- **WHEN** ACP `handle_prompt()` emits `UserMessageChunk` with `message_id="msg_123"` and registers it in the shared dedup set
- **AND** `UserMessageInsertedEvent(message_id="msg_123")` arrives via EventBus
- **THEN** the handler finds `"msg_123"` in the dedup set
- **AND** skips the EventBus-derived emission (no duplicate)

#### Scenario: Internal steer with no prior protocol emission
- **WHEN** `steer_from_background_task()` publishes `UserMessageInsertedEvent(message_id="msg_456")`
- **AND** no protocol handler has emitted a user message with `message_id="msg_456"`
- **THEN** the handler does not find `"msg_456"` in the dedup set
- **AND** emits the user message to the frontend
- **AND** adds `"msg_456"` to the dedup set

### Requirement: Emission helper uses logfire.span and exception handling

All `asyncio.create_task()` emission calls SHALL use a shared emission helper that wraps the emission in `logfire.span("event.user_message_inserted.emit")` to prevent orphan traces. The helper SHALL catch all exceptions with `try/except Exception` and log a warning, preventing task failures from propagating.

#### Scenario: Emission helper wraps in logfire.span
- **WHEN** the emission helper is called via `asyncio.create_task()`
- **THEN** the emission is wrapped in `logfire.span("event.user_message_inserted.emit", session_id=session_id)`
- **AND** any exception during emission is caught and logged as a warning

#### Scenario: Emission helper handles EventBus=None
- **WHEN** the emission helper is called and no EventBus is available
- **THEN** no publish is attempted
- **AND** no exception is raised
